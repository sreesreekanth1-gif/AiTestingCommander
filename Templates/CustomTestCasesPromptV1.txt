## ANTI-HALLUCINATION PROTOCOL (MANDATORY — READ FIRST)
============================================================
You MUST follow these rules. Violating any of them invalidates the output.

1. **Source-Grounded Only**: Every test case must trace back to a specific, quoted snippet 
   from the user story, acceptance criteria, or supporting artifacts. If you cannot quote 
   the source, you cannot write the test.

2. **No Invented Artifacts**: Do NOT invent field names, API endpoints, error codes, 
   status values, business rules, UI elements, validation limits, or user roles that 
   are not present in the input. If a value is needed but not given (e.g., "max length"), 
   mark it as `[ASSUMPTION — NEEDS CONFIRMATION]` and explain the reasoning.

3. **Explicit vs. Inferred vs. Assumed**: Every test case must be tagged with one of:
   - `EXPLICIT` — directly stated in the source
   - `INFERRED` — logically derivable from stated requirements (explain the inference)
   - `ASSUMED` — based on industry standard / best practice (flag for review)

4. **Ambiguity Surfacing**: Before writing tests, list every ambiguity, missing detail, 
   or contradiction in a "Clarifications Needed" section. Do not silently resolve them.

5. **No Generic Filler**: Do not produce boilerplate test cases that don't map to the 
   actual story (e.g., "Test that the system is up" unless uptime is in scope).

6. **Traceability is Mandatory**: Each test case includes a `Source Reference` field 
   citing the exact AC ID or quoted line it validates.

7. **Self-Verification Pass**: After generating tests, run the Self-Check Protocol 
   (defined below) and report results.

## PHASE 1 — PRE-ANALYSIS (perform before writing any test case)
============================================================
Output the following before generating tests:

A. **Story Decomposition**: List every distinct functional requirement, business rule, 
   actor, input, output, state, and dependency you extracted — each with a direct quote 
   from the source.

B. **Clarifications Needed**: List every ambiguity, missing value, or contradiction. 
   For each, propose a reasonable default and mark dependent tests as `ASSUMED`.

C. **In-Scope vs. Out-of-Scope**: Explicitly declare what this story covers and what 
   it does NOT cover (so we don't fabricate tests for adjacent functionality).

D. **Applicability Matrix**: For each of the coverage dimensions below, mark 
   `APPLICABLE` / `NOT APPLICABLE` with a one-line justification grounded in the source. 
   This prevents writing security tests for a story that has no security surface, etc.

## PHASE 2 — COVERAGE DIMENSIONS
============================================================
Generate test cases across ALL applicable dimensions. Skip non-applicable ones 
(but justify the skip in Phase 1D).

### 1. FUNCTIONAL COVERAGE
- Happy path / main success scenarios
- Alternative valid paths
- Exception paths and edge cases
- Business rule validation (each rule tested independently and in combination)

### 2. INPUT VALIDATION COVERAGE
- Valid inputs (positive cases)
- Invalid inputs (negative cases)
- Boundary values: min, min-1, min+1, max, max-1, max+1, nominal
- Special characters, Unicode, emoji, RTL text
- Empty / null / undefined / zero / whitespace-only values
- Type mismatches (string where number expected, etc.)
- Length constraints (too short, too long, exact)
- Format variations (date formats, number locales, encodings)
- Leading/trailing whitespace, case sensitivity

### 3. STATE & TRANSITION COVERAGE
- Initial state validation
- All valid state transitions
- All invalid / forbidden state transitions
- Final state validation
- Idempotency (repeated actions in same state)

### 4. DATA COVERAGE
- Equivalence partitions (one test per partition)
- Boundary data
- Data combinations / pairwise where multiple inputs interact
- Realistic vs. synthetic data
- Data volume scenarios (empty set, single record, many records)

### 5. BUSINESS LOGIC COVERAGE
- Each decision point (if/then/else, switch branches)
- All conditional branches (branch coverage)
- Loop scenarios: zero, one, many, max iterations
- Calculations and formulas (with explicit expected values)
- Rounding, currency precision, time zone handling (if applicable)

### 6. INTEGRATION COVERAGE
- Component-to-component interactions
- Upstream / downstream data flow
- API contracts: request schema, response schema, status codes
- External dependency failure modes (timeout, 5xx, malformed response)
- Contract / consumer-driven scenarios

### 7. ERROR & EXCEPTION COVERAGE
- Expected error scenarios (validation, business rule violation)
- Unexpected errors (system failure, dependency down)
- Error message content, code, and localization
- Recovery and rollback mechanisms
- Retry behavior
- Timeout scenarios

### 8. PERFORMANCE & LOAD COVERAGE (if applicable)
- Normal load
- Peak load
- Stress / soak / spike scenarios
- Response time SLAs
- Concurrency under load

### 9. SECURITY COVERAGE (if applicable)
- Authentication flows (valid, invalid, expired, locked)
- Authorization / RBAC (each role × each action)
- Session management (expiry, hijack, fixation)
- Input sanitization: SQLi, XSS, command injection, path traversal, SSRF
- Sensitive data exposure (PII in logs, responses, URLs)
- CSRF, clickjacking, security headers
- Rate limiting and brute-force protection

### 10. UI/UX COVERAGE (if applicable)
- Element visibility, enabled/disabled, focus states
- User interaction flows
- Responsive behavior (breakpoints)
- Navigation and back-button behavior
- Form submission: valid, invalid, partial, double-submit
- Loading, empty, and error states

### 11. ACCESSIBILITY COVERAGE (NEW — if UI involved)
- Keyboard-only navigation
- Screen reader / ARIA labels
- Color contrast (WCAG AA/AAA)
- Focus management and skip links
- Alt text and semantic HTML
- Zoom / text resizing

### 12. LOCALIZATION & INTERNATIONALIZATION (NEW — if applicable)
- Translated strings render correctly
- Date, time, number, currency formatting per locale
- RTL languages
- Character set / encoding (UTF-8, multi-byte)
- Pluralization and gender rules

### 13. COMPATIBILITY COVERAGE (NEW — if applicable)
- Browser matrix (specify which)
- OS / device matrix
- Mobile vs. desktop vs. tablet
- Screen sizes / orientations
- Backward compatibility with prior versions / API versions

### 14. CONCURRENCY & RACE CONDITIONS (NEW — if applicable)
- Simultaneous edits / writes
- Optimistic vs. pessimistic locking
- Double-submit / replay protection
- Dirty reads, lost updates

### 15. PERSISTENCE & DATA INTEGRITY (NEW — if applicable)
- Data correctly stored and retrieved
- Transaction commit / rollback
- Referential integrity
- Soft-delete vs. hard-delete behavior
- Migration / schema-change compatibility

### 16. NETWORK & CONNECTIVITY (NEW — if applicable)
- Offline behavior
- Slow / flaky network
- Connection drop mid-operation
- Sync after reconnect

### 17. OBSERVABILITY (NEW — if applicable)
- Logs emitted with correct severity and content
- Metrics / counters incremented
- Traces / correlation IDs propagated
- Audit trail entries created

### 18. COMPLIANCE & REGULATORY (NEW — if applicable)
- GDPR / CCPA: consent, data export, right to deletion
- HIPAA / PCI-DSS / SOC2 controls (if relevant domain)
- Data retention and purging
- Geographic data residency

### 19. NOTIFICATIONS & SIDE EFFECTS (NEW — if applicable)
- Emails / SMS / push notifications triggered correctly (and only when intended)
- Webhook / event-bus emissions
- Analytics events fired with correct payload

### 20. FEATURE FLAGS & CONFIGURATION (NEW — if applicable)
- Behavior with flag ON
- Behavior with flag OFF
- Mid-session flag flip behavior
- Config override precedence

## PHASE 4 — VERIFICATION & TRACEABILITY
============================================================
Provide the following artifacts after the test cases:

   more TC IDs. Flag any AC with zero tests as a GAP.

   - # of test cases
   - Coverage % (with definition of how % was calculated)

   - Tests deferred pending clarification

5. **Test Effort Estimate**: 
   - Manual execution hours

6. **Duplication Check**: Confirm no two test cases have the same steps + expected result

## PHASE 5 — SELF-CHECK PROTOCOL (run before finalizing)
============================================================
Output a YES/NO checklist with brief evidence for each:

[ ] Every test case has a Source Reference quoting the input
[ ] No test case introduces fields, endpoints, roles, or rules not in the source
[ ] Every ASSUMED test case is flagged and listed in Clarifications Needed
[ ] Every applicable coverage dimension has at least one test (or a justified skip)
[ ] No two test cases are functional duplicates
[ ] Every AC has at least one test (or is flagged as a gap)

[ ] All test data is concrete (no placeholders like "valid input")
[ ] Boundary tests include min-1, min, min+1, max-1, max, max+1 where bounds exist
[ ] Negative tests exist for every positive test of a validated input
[ ] Priorities are justified, not arbitrary

If any item is NO, fix it before returning the final output.
